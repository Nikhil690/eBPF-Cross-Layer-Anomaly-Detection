# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build Commands

```bash
# Generate BPF skeletons + compile binary
make build          # runs go generate then go build -o ebpf-cla .

# Individual steps
make generate       # go generate ./... (bpf2go: compiles xdp.c, tc.c, tracepoint.c)
make vmlinux        # regenerate bpf/vmlinux.h from running kernel

# Run (requires root)
sudo ./ebpf-cla ens3
sudo ./ebpf-cla ens3 --csv results/raw/out.csv --label benign

# Clean generated artifacts
make clean          # removes *_bpfel.go *_bpfel.o *_bpfeb.go *_bpfeb.o ebpf-cla
```

There are no unit tests in this codebase. Evaluation is done via the scripts in `results/`.

## Evaluation

# Full labelled dataset collection (~5 min)

## Architecture

This is an eBPF-based intrusion detection system that correlates events from three kernel layers simultaneously. The core insight is using `socket_cookie` (`bpf_get_socket_cookie()`) as a zero-copy join key that links events across layers without userspace intermediary.

### Data Flow

```
XDP ingress (xdp.c)           → flow_stats_map (LRU_HASH, 5-tuple key)
TC egress (tc.c)               → reads flow_stats_map via reversed 5-tuple
                               → writes corr_window_map (cookie key)
                               → emits to ring_events when window age > 5ms
Tracepoint sys_enter_connect  → writes proc_state_map (PID key)
(tracepoint.c)                → updates corr_window_map with proc stats

Go userspace (main.go):
  - ring buffer reader goroutine  → reads ring_events → processRecord()
  - map sweeper goroutine (20ms)  → iterates corr_window_map, emits stale entries
                                    (needed for short-lived flows like port scans)
  → CorrRecord → ExtractFeatures() → OnlineStats.Update() → AnomalyScore()
  → CSV output and/or ALERT log line
```

### Go Source Files

- **`loader.go`** — eBPF lifecycle: loads objects, pins map sharing between programs (XDP→TC shares `flow_stats_map`, TC→TP shares `corr_window_map` + `ring_events`), attaches XDP/TCX/tracepoint hooks, starts 20ms sweeper goroutine.
- **`correlator.go`** — `CorrRecord` struct (mirrors `bpf/common.h`), `ParseCorrRecord` (ring buffer bytes → struct), `tcRecordToCorrRecord` (bpf2go type → canonical type), `ExtractFeatures` (CorrRecord → 9-feature FeatureVector).
- **`detector.go`** — `OnlineStats` (Welford's algorithm, mutex-protected), `AnomalyScore` (Euclidean z-score, threshold=4.0, warmup guard n<50).
- **`main.go`** — entry point, CLI flag parsing (`<iface>`, `--csv`, `--label`), CSV writer, two consumer goroutines, signal handler.

### BPF Source Files (`bpf/`)

- **`common.h`** — shared structs (`cla_flow_key`, `cla_flow_stats`, `cla_corr_record`, `cla_proc_state`), all map definitions, layer coverage bit constants (LAYER_XDP=0x01, LAYER_TC=0x02, LAYER_SYSCALL=0x04, LAYER_UPROBE=0x08).
- **`xdp.c`** — XDP ingress hook: parses Ethernet/IP/TCP headers, updates `flow_stats_map` per 5-tuple.
- **`tc.c`** — TC egress hook: gets `socket_cookie`, reverse-lookups `flow_stats_map`, assembles `corr_window_map` entry, flushes to `ring_events` when window is complete.
- **`tracepoint.c`** — `sys_enter_connect` tracepoint: updates `proc_state_map` and merges into `corr_window_map`.

### Generated Files

`*_bpfel.go` / `*_bpfel.o` are generated by `bpf2go` (via `go generate`) from the BPF C sources. Do not edit them manually. The `*_bpfeb.*` variants are big-endian (unused on x86).

## Requirements

- Linux kernel ≥ 5.8 (ring buffer API)
- clang ≥ 14 (BTF support)
- Go ≥ 1.21
- libbpf-dev (`bpf_helpers.h`)
- Root / `CAP_BPF` + `CAP_NET_ADMIN` + `CAP_PERFMON` to run

Tested on Linux 6.8.0-106-generic, clang 18, Go 1.25.

## Known Limitations

- **`proc_state` rarely joins**: `skc_cookie` is 0 at tracepoint time (socket hasn't traversed TC yet). The syscall layer (`S` in layer mask) rarely appears in practice.
- **`LAYER_UPROBE` (0x08) is reserved** but not implemented — TLS uprobe on `libssl.so` is designed but not built.
- **Online stats reset on restart** — Welford state is in-memory; warmup period repeats after each restart.
- **sklearn models have low recall** — benign short connections overlap with attack short connections at per-flow level; the rule-based detector (F1=0.957) outperforms all ML models.
